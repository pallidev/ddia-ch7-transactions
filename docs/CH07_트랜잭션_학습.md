## 0. 트랜잭션이 왜 필요한가 — 한 줄 요약

데이터 시스템에서는 항상 무언가가 고장난다: 디스크 풀, 프로세스 크래시, 네트워스 끊김, **동시에 들어오는 여러 클라이언트의 충돌**.
트랜잭션은 여러 읽기/쓰기를 **하나의 논리적 단위**로 묶어, "전부 성공하거나(commit) 전부 실패(abort/rollback)한다" 는 보장을 준다.
핵심은 **부분 실패(partial failure) 를 앱이 신경 쓰지 않아도 되게 만드는 것**이다.

---

## 1. ACID — 네 글자의 (불완전한) 약속

| 글자 | 의미 | 누구 책임? | 이 프로젝트에서 보는 곳 |
|---|---|---|---|
| **A**tomicity (원자성) | 중간에 고장 나면 **지금까지 한 쓰기도 전부 취소** (all-or-nothing). "abortability" 가 더 정확한 이름. | DB | `AtomicityTest` |
| **C**onsistency (일관성) | "데이터가 항상 옳은 상태" 라는 **앱 정의 불변식**. DB가 강제 불가 — 앱이 올바른 트랜잭션을 짜야 함. | **앱** | (개념만 — `Doctor` 의 "1명 이상 당직", `Booking` 의 "중복 예약 금지") |
| **I**solation (격리) | 동시에 실행되는 트랜잭션이 **서로 간섭하지 않는다**. | DB | 나머지 모든 동시성 테스트 |
| **D**urability (지속성) | commit 된 데이터는 고장 나도 사라지지 않는다 (디스크/복제). | DB | (단일 노드 인메모리 테스트라 다루지 않음) |

> 책의 핵심 통찰: **C는 사실 ACID에 속하지 않는다.** 일관성은 앱의 책임이지 DB가 주는 것이 아니다.
> A, I, D는 DB의 속성이지만 C는 앱의 속성이다. 이 장의 나머지는 **I(격리)** 에 대한 깊은 탐구다.

## 2. 격리 수준과 race condition — 이 장의 핵심

격리 수준은 **"어떤 동시성 버그까지 막아주는가"** 의 스펙트럼이다. 약한 수준일수록 빠르지만 버그가 많다.

```
  약하다(빠름) ──────────────────────────────────────► 강하다(느림)
  Read Uncommitted → Read Committed → Repeatable Read(snapshot) → Serializable
```

아래 표는 책이 다루는 6가지 race condition 을 **어떤 격리 수준이 막는지** 정리한 것이다.

| Race condition | 한 줄 설명 | Read Committed | Repeatable Read | 이 프로젝트 테스트 |
|---|---|---|---|---|
| **Dirty read** | 남의 *커밋 안 된* 쓰기를 읽음 | ✅ 막음 | ✅ 막음 | — (H2 한계로 생략) |
| **Dirty write** | 남의 *커밋 안 된* 쓰기를 덮어씀 | ✅ 막음 | ✅ 막음 | — |
| **Read skew** (nonrepeatable read) | 같은 행을 두 번 읽었는데 값이 달라짐 | ❌ 발생 | ✅ 막음 | `ReadSkewTest` |
| **Lost update** (갱신 손실) | 동시 read-modify-write 로 한 쪽 수정이 묵살 | ❌ 발생 | ❌ 발생 | `LostUpdateTest` |
| **Write skew** (쓰기 비대칭) | 같은 데이터 읽고 결정한 뒤 *서로 다른 행* 을 고쳐 불변식 위반 | ❌ | ❌ | `WriteSkewTest` ⭐ |
| **Phantom** (팬텀) | 검색 결과를 바꾸는 쓰기(주로 INSERT) 가 끼어듦 | ❌ | ❌ | `PhantomTest` |

> **가장 중요한 교훈: Repeatable Read(snapshot isolation) 조차 lost update, write skew, phantom 을 못 막는다.
> 이 셋을 **모두** 막는 유일한 격리 수준은 Serializable 뿐이다.**

---

### 2.1 Read skew (nonrepeatable read) → `ReadSkewTest`

**시나리오**(책 Fig 7-6): Alice 가 두 계좌 잔액을 조회하는 사이 이체 트랜잭션이 하나를 바꾼다.
한 트랜잭션이 같은 행을 **두 번** 읽었을 때 값이 바뀌면 read skew.

```
READ COMMITTED:  첫 읽기=100 ──(T1이 200으로 커밋)── 둘째 읽기=200  → 값이 바뀜 ❌
REPEATABLE READ: 첫 읽기=100 ──(T1이 200으로 커밋)── 둘째 읽기=100  → 스냅샷 일관 ✅
```

실행 결과(`ReadSkewTest`):
- READ COMMITTED: `first=100, second=200` → nonrepeatable read 관찰
- REPEATABLE READ: `first=100, second=100` → snapshot 으로 일관

> **MVCC(다중 버전 동시성 제어)**: Repeatable Read 가 가능한 이유. DB가 행의 여러 버전을 유지해서,
> 각 트랜잭션이 시작 시점의 "일관된 스냅샷" 만 본다. readers never block writers, writers never block readers.
> 자세한 구현(created_by/deleted_by 버전 관리) 은 책 Fig 7-7 참조.

### 2.2 Lost update (갱신 손실) → `LostUpdateTest` ⭐ 책의 첫 예시(Fig 7-1)

**시나리오**: 두 클라이언트가 카운터를 동시에 +1. 42 → 44 가 되어야 하는데 43 이 된다.

```
T1: read(0) ─────────────── write(0+1 = 1)
T2:          read(0) ─────────────────── write(0+1 = 1)   ← T1의 +1 이 묵살됨
결과: 1 (2가 되어야 함)
```

이 프로젝트 테스트 출력:
```
[T1] 읽은 값 = 0
[T2] 읽은 값 = 0        ← 둘 다 같은 "0" 을 읽음
[결과] 최종 잔액 = 1    ← 2가 되어야 하지만 1. 한 증가가 손실.
```

**ORM 주의**: JPA/Hibernate 로 엔티티를 읽어 `balance += 1` 후 `save()` 하면 기본적으로 이 안전하지 않은 패턴이 된다. 책 p.243 "ORM 은 unsafe read-modify-write 를 쓰기 쉽게 만든다."

**해결책 3가지** (모두 이 프로젝트에 구현됨):

| 방법 | 코드 | 원리 |
|---|---|---|
| **원자 갱신** | `CounterService.incrementAtomic` → `AccountRepository.addBalance` (벌크 `UPDATE balance = balance + ?`) | read-modify-write 를 SQL 한 줄로. DB가 행 잠금으로 직렬화 |
| **비관적 락** | `PessimisticLockService.incrementWithLock` → `findWithLockingById` (`SELECT ... FOR UPDATE`) | 읽을 때부터 행을 잠가서 두 번째 트랜잭션이 대기 |
| **낙관적 락** | `OptimisticLockService` → `Account.version` (`@Version`) | 잠그지 않고 진행, commit 시 버전 검사 → 진 사람 abort & 재시도 |

실행 결과(해결책):
```
[결과] 최종 잔액 = 2   ← 원자 갱신으로 두 +1 이 모두 반영
```

### 2.3 Write skew (쓰기 비대칭) → `WriteSkewTest` ⭐⭐ 책의 가장 중요한 예시(Fig 7-8)

**시나리오**: 병원 당직. "한 shift 에는 항상 1명 이상 당직 의사가 있어야 한다."
Alice 와 Bob 이 동시에 당직을 포기한다.

```
T1(Alice): count(당직)=2 → "한 명 빠져도 되지" → Alice off
T2(Bob):   count(당직)=2 → "한 명 빠져도 되지" → Bob off
결과: 당직 0명  ← 불변식 위반!
```

이 프로젝트 테스트 출력:
```
[T1/Alice] 당직 인원 = 2 → 포기 결정
[T2/Bob]   당직 인원 = 2 → 포기 결정
[결과] 최종 당직 인원 = 0 (불변식: 항상 1명 이상이어야 함)
```

**왜 위험한가?** 두 트랜잭션이 **서로 다른 행**(Alice 행, Bob 행) 을 고친다.
그래서 갱신 손실도 더러운 쓰기도 아니다 → Repeatable Read 가 전혀 못 막는다.

**이 패턴이 숨어있는 곳들**(책 p.249): 회의실 중복 예약, 유저명 중복 가입, 게임 말 겹침, 포인트 이중 지출.
공통점 = **"검사(SELECT) → 결정 → 쓰기"** 흐름.

**해결책**:
- ✅ **Serializable 격리 수준** — 유일하게 모든 write skew 를 막는다.
- ✅ **명시적 잠금(`SELECT ... FOR UPDATE`)** — 검사 대상 행을 잠가 직렬화. 이 프로젝트의 `OnCallService.goOffCallWithLock`.
- ⚠️ 원자 단일 객체 연산 / 자동 갱신 손실 감지 — **도움 안 됨**(여러 객체가 얽힘).

실행 결과(해결책):
```
[결과] 최종 당직 인원 = 1   ← FOR UPDATE 로 한 명은 포기 거부되어 불변식 보존
```

### 2.4 Phantom (팬텀) → `PhantomTest` (책 Example 7-2)

**시나리오**: 회의실 예약. "같은 방, 겹치는 시간엔 2개 예약 금지."
흐름: (1) 겹치는 기존 예약 검색 (2) 없으면 INSERT.

```
T1: 검색(겹침 없음) ─────────── INSERT
T2:  검색(겹침 없음) ─────────── INSERT   ← T1 이 아직 커밋 전이라 "없음" 으로 보임
결과: 같은 시간에 예약 2개
```

**팬텀이 까다로운 이유**: 잠글(LOCK) 대상 **행이 아직 존재하지 않는다**(INSERT 로 새로 생김).
그래서 `SELECT ... FOR UPDATE` 로는 잠을 수가 없다.

**해결책**(이 프로젝트에서 가장 실용적인 것을 채택):
- ✅ **유일 제약(unique constraint)** on `(roomId, startTime)` — `Booking` 엔티티에 선언.
  검사-후-쓰기가 팬텀에 뚫려도, DB 제약이 동시 INSERT 중 하나를 abort 시킨다(= **충돌의 구체화, materializing conflicts** 와 같은 맥락).
- ✅ **Serializable 격리 수준**(이론적 정해).
- ⚠️ materializing conflicts(시간 슬롯 테이블을 미리 만들어 잠금 대상으로 삼기) — 책 p.250, "최후의 수단".

실행 결과:
```
[T1] 겹치는 예약 있나? = false
[T2] 겹치는 예약 있나? = false
[결과] 예약 수 = 1   ← 유일 제약이 중복 INSERT 하나를 막음
```

## 3. Serializable 을 어떻게 구현하는가 (3가지 알고리즘)

모든 race condition 을 막는 유일한 격리 수준. 구현 3가지:

| 알고리즘 | 원리 | 장/단 | 대표 |
|---|---|---|---|
| **실제 직렬 실행** | 한 스레드에서 트랜잭션을 한 번에 하나씩. in-memory + 짧은 stored procedure 전제. | 가장 단순, 단일 코어 한계 | VoltDB, Redis, Datomic |
| **2단계 잠금(2PL)** | 읽기=공유 잠금, 쓰기=배타 잠금. writer 가 reader 도 막음. | 완전한 직렬화, **느리고 교착 잦음** | MySQL/SQLServer 의 SERIALIZABLE |
| **직렬화 가능 스냅샷 겈리(SSI)** | 낙관적. 스냅샷으로 읽되, commit 시 "내가 읽은 전제(premise) 가 무효화됐는가" 검사 → 문제면 abort. | 비교적 빠름, 경합 높으면 재시도 비용 | PostgreSQL 9.1+ 의 SERIALIZABLE |

> **2PL ≠ 2PC**. 2PL(2단계 잠금)은 직렬화 격리 알고리즘, 2PC(2단계 커밋)는 분산 트랜잭션 커밋 프로토콜(9장). 완전 다른 것.

**SSI 의 핵심 통찰**(p.262): write skew 는 결국 **"오래된 전제(outdated premise) 에 기반한 결정"** 이다.
"당직이 2명이다" 라고 읽었는데 커밋할 때 그 전제가 이미 거짓이 됐다 → SSI 는 이걸 감지해 abort 한다.

## 6. H2 vs 실제 MySQL — 주의사항

이 프로젝트의 테스트는 **H2(MySQL 호환 모드)** 로 돈다. 개념 학습에는 충분하지만, 실제 DB와 **미묘한 차이**가 있다:

- **H2 2.x는 MVCC 만 지원** → `READ UNCOMMITTED` 격리 수준이어도 dirty read 가 발생하지 않는다. 그래서 dirty read 예시는 생략했다.
- 격리 수준별 race condition 방지 여부는 **DB 구현마다 다르다**(책 p.233의 핵심 메시지). 예를 들어 MySQL InnoDB 의 Repeatable Read는 lost update 를 일부 막지만, PostgreSQL/Oracle 은 다르다.
- **실제 프로덕션 동작은 반드시 타깃 DB(MySQL)에서 직접 검증**해야 한다. 이 테스트는 "개념을 체감하는" 용도다.

> 운영(`main`) 설정은 그대로 **MySQL** 이다(`application.properties` + `mysql-connector-j`).
> 테스트만 `src/test/resources/application.yml` 에서 H2 로 덮어쓴다.

## 7. 핵심 요약 (암기 포인트)

1. **ACID 의 C는 사실 앱 책임**이다. DB가 주는 건 A, I, D.
2. **Repeatable Read(snapshot)는 lost update, write skew, phantom 을 못 막는다.**
3. **write skew = "검사 → 결정 → 쓰기" 패턴의 동시성 버그.** 가장 은밀하고 위험하다.
4. **모든 race condition 을 막는 건 Serializable 하나뿐.** 그 외는 명시적 잠금/원자 갱신/제약 으로 앱이 보완해야 한다.
5. 해결책 선택: 경합 낮으면 **낙관적 락(@Version)**, 단순 증가는 **원자 갱신**, 복잡한 검사-후-쓰기는 **FOR UPDATE** 또는 **Serializable**.
